> Документ собран из **фактического кода** репозитория (ветка `main`), дизайн-документов
> `docs/design/*`, плана `docs/plan-college-bot-mvp.md` и опросника поддержки.

> **«ПРЕДПОЛОЖЕНИЕ»**. Реальные имена статусов, полей и файлов сохранены как в коде.

## 1. Назначение проекта

**Проблема.** Лиды приходят с рекламы (Instagram) в WhatsApp; пишут в основном **родители
абитуриентов** (и сами школьники). Операторы приёмной комиссии не успевают отвечать: большие
задержки (вопрос ночью — ответ утром), ответы разных операторов несогласованы. Часть клиентов
теряется из-за тишины.

**Кто пользуется ботом (внутри):** менеджеры/операторы приёмной комиссии колледжа
**Intellect IT & Business College** (Бишкек, ул. Ибраимова 103/1А, B Block).

**Кто пишет боту (снаружи):** родители абитуриентов и школьники. Языки — **русский и
кыргызский** (в исходных переписках заметная доля кыргызского). Бот определяет язык и
отвечает на языке клиента.

**Ожидаемый результат работы бота:**
- консультировать 24/7 **только фактами из базы знаний**;
- квалифицировать абитуриента (база 9/11 класс, имя, направление);
- довести до целевого действия — **запись на вступительный тест** или **передача менеджеру**;
- вести карточку лида по канбану, уведомлять менеджера, автодожимать «молчащих».

**ЖЕЛЕЗНОЕ ПРАВИЛО (сквозное требование заказчика):** бот **НИКОГДА не врёт и не выдумывает**.
Не знает ответ / вопрос спорный → **сразу эскалация на живого менеджера**. Реализовано двумя
слоями (промпт+FAQ-whitelist и пост-валидатор) — см. §7.

**Роль сотрудников приёмной комиссии:** принимают эскалированные диалоги, ведут переписку
вручную через админ-панель (перехват), подтверждают дату/формат теста, обсуждают оплату/
скидки/договор (бот это не делает), выставляют финальный исход карточки (Поступает/Слив).

**Роль CRM / доски с карточками:** канбан-доска в **собственной админ-панели проекта**
(FastAPI + HTMX, встроена в этот же сервис) — источник рабочего процесса менеджера: колонки-
статусы, чат с клиентом, кнопки перехвата, аналитика, редактор FAQ. Дополнительно
предусмотрена интеграция с **внешней админкой заказчика (Emir) через вебхуки** (адаптер
`CollegeAdminCrm`, пока заглушка — см. §7 и §14). Заказчик упоминал Trello + Bitrix как
существующие инструменты колледжа; в код они как активные интеграции НЕ включены.

## 3. Текущий процесс обработки лида

Пошагово, как это делает код (`app/core/orchestrator.py`, `app/funnels/admission.py`,
`app/agent/runner.py`). «Стадия» — `DialogState.stage`; «колонка» — колонка канбана
(маппинг `STAGE_TO_COLUMN`, `app/core/leadstate.py`).

1. **Входящее сообщение** приходит вебхуком провайдера → маршрутизируется к боту по
   `profile_id` (Wappi) / `BOT_ID` (Bitrix) / bot_id (Telegram). Дедуп по id события.
   - Клиент: «Здравствуйте, расскажите про поступление».
   - Данные: номер (`user_id`), текст, канал, chat_id.
   - Стадия старта: `greeting` (колонка **«Новые лиды»**).

2. **Лог входящего** пишется в панель **всегда** (даже при перехвате/выключенном боте) —
   менеджер видит реплику вживую (`orchestrator._log_in`).

3. **Проверка рубильника и перехвата.** Если авто-ответы выключены (`bots_enabled=false`)
   или диалог перехвачен менеджером — бот молчит (дальше отвечает человек). Иначе — работает.

4. **FAQ-слой (детерминированный).** До LLM пробуется совпадение по whitelist-паттернам
   (`app/core/faq.py`). Если правило найдено — мгновенный готовый ответ; правило типа
   `handoff_only` (напр. «часы работы», «проходной балл») дополнительно переводит диалог
   менеджеру (`stage=manager`, `intercepted=true`).

5. **Воронка admission (LLM).** Если FAQ не сработал — `run_admission_turn`: system-промпт
   персоны «Айдана» + база знаний + правила языка/эскалации, 4 инструмента. LLM:
   - отвечает фактом из базы знаний;
   - собирает квалификацию через `ask_qualification` (порядок: **база 9/11 → имя →
     направление**), 1 вопрос за ход, сначала ответ — потом вопрос;
   - стадия → `qualification` (колонка **«Квалификация»**); карточка-лид создаётся в CRM
     при первом собранном поле.
   - Клиент: «После 9 класса, зовут Айбек, интересует кибербезопасность».
   - Данные карточки: `grade_base=9`, `name=Айбек`, `direction=кибербезопасность`.

6. **Горячий лид → запись на тест.** Клиент «хочу записаться на тест / поступать» →
   инструмент `escalate_to_office`. **Обязательное условие — собрано имя** (иначе бот
   сначала спросит имя, запись не подтверждает). Стадия → `test_invite` (колонка
   **«Приглашён на тест»**). Бот НЕ называет дату/время/формат/проходной балл — их
   подтверждает менеджер.

7. **Эскалация → менеджер.** Вопрос вне базы, торг о цене/скидке, оплата/договор, жалоба,
   просьба человека, неуверенность, 2 бессмысленных хода → `handoff_to_manager(reason)`.
   Стадия → `manager` (колонка **«У менеджера»**). Причина пишется в карточку
   (`escalation_reason`).

8. **Момент, когда бот прекращает авто-ответы (замолкает):**
   - при авто-хендоффе `stage=manager` бот отправляет одну прощальную реплику и ставит
     `intercepted=true` — дальше в чате отвечает только человек (решение заказчика
     23.06.2026, зафиксировано в `orchestrator.py:206-211`);
   - когда менеджер нажал «Перехватить» или сам ответил из панели;
   - когда выключен рубильник `bots_enabled`.
   После авто-хендоффа, пока менеджер молчит, бот **один раз** честно подтверждает клиенту
   ожидание (`wait_ack`, `orchestrator._maybe_wait_ack`), чтобы клиент не висел в тишине.

9. **Ответ клиенту** уходит через канал; статус доставки (✓/✓✓) обновляется в панели по
   webhook’ам delivery-status (Wappi).

10. **Закрытие лида.** Финальный исход выставляет **менеджер вручную** кнопкой:
    `won` («🎓 Поступает») или `lost` («❌ Слив»); ручной исход бот авто-синхронизацией не
    перетирает. Молчащих ведёт автодожим (§10), но он выключен по умолчанию.

> Момент создания задачи сотруднику как отдельной «task»-сущности в CRM — **не определено**
> (в текущем коде «задача» = появление карточки в колонке «У менеджера»/«Приглашён на тест»
> + фоновые алерты о ждущем клиенте, а не запись в трекере задач).

## 4. Статусы карточек

> ⚠ **Важно для приёмника проекта:** набор статусов ниже — из **текущего кода и дизайна**
> (Fable), а НЕ из подтверждённой заказчиком воронки. **Блок 8 опросника (статусы Bitrix/
> Trello колледжа) пуст** — реальные статусы заказчика не собраны. Приведи к ним после
> подтверждения.

Есть два уровня: **стадия** (`DialogState.stage`, внутренняя) и **колонка канбана**
(`BOARD_COLUMNS` в `app/admin/router.py`). Маппинг — `STAGE_TO_COLUMN` (`leadstate.py:12-19`).

**Колонки канбана (BOARD_COLUMNS):**

| Ключ | Ярлык | Назначение | Условие попадания | Кто ставит | Задача? | Бот продолжает? |
|---|---|---|---|---|---|---|
| `greeting` | Новые лиды | стартовое состояние | первое сообщение (`stage` greeting/new) | бот (авто) | нет | да |
| `qualification` | Квалификация | собираем имя/базу/направление | вызван `ask_qualification` | бот | нет | да |
| `progress` | Консультация | идёт консультация | `stage` consulting/progress (`crm_update_stage`) | бот | нет | да |
| `office` | Приглашён на тест | горячий лид, зовём на тест | `escalate_to_office` (`stage` test_invite/office) | бот и менеджер | да (менеджер подтверждает тест) | нет (бот замолкает при handoff-логике) |
| `manager` | У менеджера | эскалация к человеку | `handoff_to_manager` или ручной перехват | бот и менеджер | да | нет (бот молчит) |
| `silent` | Молчат (на дожим) | вычисляемая, не хранится | `is_silent()` истинно (тишина > порога) | вычисляется | — | зависит (дожим, если включён) |
| `follow_up` | Повторное касание | ушёл автодожим-пинг | `followup` job (`stage=follow_up`, `followup_sent=true`) | джоба (авто) | нет | нет (ждём ответа клиента) |

- **Допустимые следующие статусы:** drag-and-drop переносит карточку в стадию с тем же
  ключом (`COLUMN_TO_STAGE`, любая колонка кроме `silent` — она вычисляемая, руками
  недоступна). Явной машины «разрешённых переходов» нет — менеджер может двигать свободно.
- **Напоминание/следующий контакт по статусу:** явных per-статус SLA нет; тайминги —
  глобальные пороги фоновых джоб (см. §10): `alert_awaiting_minutes=10`,
  `followup_after_hours=24`, `noise_stale_days=3`.

**Исходы карточки (OUTCOMES, отдельно от колонок):**

| Ключ | Ярлык | Кто ставит | Защита |
|---|---|---|---|
| `won` | 🎓 Поступает | менеджер (вручную) | «ручной» — бот не перетирает |
| `office` | 📝 Пришёл на тест | менеджер / авто | авто-исход |
| `lost` | ❌ Слив | менеджер (вручную) | «ручной» — бот не перетирает |
| `manager` | (авто) У менеджера | авто (`_auto_outcome`) | авто |
| `in_progress` | (авто) В работе | авто | авто |

Механика защиты: `_AUTO_OUTCOMES={in_progress,office,manager}`, `_MANUAL_OUTCOMES={won,lost}`;
`_is_auto_downgrade` не даёт авто-синку затереть ручной финал (`store.py:61-67,157,321`).

Два источника полей: **`ConversationView`** (карточка панели, `store.py:36-58`) и
**`DialogState.qualification`** (собранные ботом данные, `state.py`). Ниже — фактические поля.

**Собираемая ботом квалификация (`qualification`, `REQUIRED_FIELDS=[grade_base, name, direction]`):**

| Поле | Тип | Обязательно | Кто заполняет | Источник | Меняется? | Подтверждать у клиента? |
|---|---|---|---|---|---|---|
| `name` (имя абитуриента) | строка | да | бот | реплика клиента | да | нет (спец. подтверждения нет; при родителе — имя будущего студента) |
| `grade_base` (база: 9/11) | «9»/«11» | да | бот | реплика клиента | да | нет |
| `direction` (направление) | строка | да | бот | реплика клиента / «не определился» | да | нет |
| `visit_time` (удобное время теста) | строка | нет | бот | если клиент сам сказал | да | нет |
| `escalation_reason` (причина эскалации) | строка | нет | бот | аргумент `handoff_to_manager` | да | нет |

| `stage` | строка | бот/менеджер | текущая стадия |

**Поля из запроса заказчика, которых в коде НЕТ (не определено / не реализовано):**
имя родителя отдельно от абитуриента; возраст/год рождения (в опроснике «⚠ нужно?»);
источник рекламы/кампания (см. §11); язык общения как явное поле (определяется на лету,
отдельно не хранится); «готовность к звонку»; дата/время звонка (`callback` есть как
стадия-алиас, но поля времени звонка нет); дата/время визита (есть только `visit_time`
как свободный текст); «ответственный сотрудник» = `assigned_to`; «последний/следующий
контакт» = `last_message_at` (последний) + пороги джоб (следующий, неявно); «причина отказа»
отдельно — нет (есть только `outcome=lost`); «комментарий» менеджера как поле — не определено
(есть аудит-лог и сообщения). **«статус управления ботом»** = `intercepted` + флаги
`bots_enabled`/`bots_enabled:<bot_id>`.

## 6. Сценарии диалогов

Эталонные реплики и правила — `docs/design/02-dialog-ux.md`; факты — `knowledge.py`;
детерминированные ответы — `faq.py::seed_defaults`. Формат: намерение → нужные данные →
ответ бота → стадия/колонка → условие передачи → следующий шаг.

| Сценарий | Намерение | Нужные данные | Ответ бота | Стадия | Передача менеджеру | Следующий шаг |
|---|---|---|---|---|---|---|
| **Поступление (общий)** | узнать про поступление | база 9/11 | приветствие + вопрос про базу | greeting→qualification | нет | собрать имя/направление |
| **Специальности** | узнать направления | — | список 8 направлений (единственное «длинное» сообщение) | qualification | если просят описание конкретного направления (нет в базе) → да | какое ближе? |
| **Стоимость** | узнать цену | — | «по контракту 6500 долларов; персональная скидка после теста» | без изм. | «за год или за весь курс?» → да | предложить тест |
| **Документы** | что нужно | — | «свидетельство о рождении; есть паспорт — достаточно паспорта» | без изм. | нет | ещё вопросы? |
| **Возраст/класс** | приём после 9/11 | — | «принимаем после 9 и 11; после 9 — 2 года 10 мес» | без изм. | срок после 11 → да (спорный факт) | какой вариант? |
| **Просьба позвонить** | хочет звонок | — | ПРЕДПОЛОЖЕНИЕ: трактуется как «нужен человек» → handoff | manager | да | менеджер свяжется |
| **Желание прийти** | готов прийти/на тест | имя | `escalate_to_office`; «заявку передал менеджеру, он подтвердит дату» | test_invite | спец-случай (office) | менеджер подтверждает |
| **Конкретная дата визита** | называет время | visit_time | фиксирует `visit_time`, дату НЕ подтверждает | test_invite | office | менеджер подтверждает |
| **Клиент думает** | берёт паузу | — | ПРЕДПОЛОЖЕНИЕ: вежливо закрыть ход; далее автодожим (если включён) | без изм. | нет | followup через 24ч |
| **«Напишите позже»** | отложить | — | **не определено** (спец-обработки нет) | без изм. | нет | автодожим |
| **Не заинтересован** | отказ | — | **не определено** (нет авто-`lost`; менеджер ставит вручную) | без изм. | нет | менеджер → lost |
| **Перестал отвечать** | молчание | — | нет ответа; попадает в `silent`, потом дожим | silent→follow_up | нет | 1 пинг |
| **Нестандартный вопрос** | вне базы | — | «хороший вопрос, передал менеджеру, ответит здесь» | manager | да | менеджер |
| **Жалоба** | недовольство/агрессия | — | одна спокойная фраза извинения + handoff | manager | да | менеджер; повторная агрессия — молчит |
| **Повторное обращение** | пишет снова | — | карточка `archived=false` при новом сообщении (всплывает) | по состоянию | по правилам | продолжение |
| **Существующий лид пишет** | тот же номер | — | тот же диалог (ключ `<bot_id>:<номер>`), история сохранена | по состоянию | по правилам | продолжение |
| **Действующий студент пишет** | не абитуриент | — | **не определено** (спец-роли «студент» нет; бот ведёт как admission) | — | вероятно handoff | — |
| **Сообщение не по теме** | оффтоп/троллинг | — | одна вежливая фраза возврата к поступлению, не спорит; 2 хода мимо → handoff | без изм./manager | после 2 бессмысленных ходов | вернуть к теме |

Точные ru/ky формулировки для всех whitelist-сценариев (цена, сроки, направления, тест,
проходной балл, документы, дедлайн, после 9/11, скидка, вуз США, адрес) и для эскалаций
(торг, оплата, жалоба, «дайте человека», горячий лид, вопрос вне базы) — в
`docs/design/02-dialog-ux.md` §4–§5.

## 7. Границы ответственности бота

### Бот может
- Приветствовать (один раз), представляться как «Айдана, приёмная комиссия» (персона).
- Отвечать **фактами из whitelist**: адрес, дедлайн (до 12 августа), 8 направлений (списком),
  предметы/длительность теста (математика+английский, 1,5 часа), документы, цена **6500
  долларов** (без периода), срок после 9 класса (2 года 10 месяцев), факт возможности
  поступления в вуз США (без обещаний).
- Квалифицировать: 1 вопрос за ход, порядок база→имя→направление, сначала ответ — потом вопрос.
- Приглашать на тест (`escalate_to_office`) — только когда собрано **имя**.
- Двигать карточку по стадиям бота (`ask_qualification`/`escalate_to_office`/
  `crm_update_stage`: qualification/consulting/test_invite).
- Определять язык (ru/ky/смешанный) и отвечать на языке клиента.
- Работать 24/7; в нерабочее время при эскалации честно предупреждать, что менеджер ответит
  в рабочее время.

### Бот не может (запрещено — промпт + валидатор)
- **Гарантировать** поступление, зачисление, грант, прохождение теста, скидку
  (детектор `admission_guarantee`, ловит и кыргызское «кепилдик»).
- Называть **любую сумму, кроме 6500** (детектор `admission_price_mismatch`, порог ≥100 ≠ 6500).
- Называть **процент/размер скидки** (детектор `admission_discount_amount`).
- Называть **проходной балл цифрой** (детектор `admission_passing_score`; «90+» — никогда).
- Утверждать **срок «3 года»** и **срок после 11 класса** (детектор `admission_duration_claim`;
  расхождение 1г10м vs «3 года» не разрешено).
- Называть **дату/время/формат теста**, период цены (за год/курс), утверждать «менеджер онлайн».
- Называть себя ботом/ИИ по своей инициативе; на «ты бот?» — не подтверждает и не отрицает.
- Ставить исход `won`/`lost` (это только менеджер).

**Правила по темам:**
- **Цены:** только 6500 долларов, без периода; торг/индивидуальная цена → менеджер.
- **Скидки:** факт «есть, после теста» — можно; размер/процент — никогда, → менеджер.
- **Обещания:** никаких гарантий — честно, без обещаний, → менеджер.
- **Документы:** свидетельство о рождении / паспорт — из whitelist, можно.
- **Персональные данные:** бот собирает только имя/базу/направление; спец-политики
  хранения/удаления ПДн — **не определено** (см. §14).
- **Конфликт:** одна спокойная фраза + handoff; повторная агрессия — молчит, остаётся у менеджера.
- **Неуверенность:** любой факт вне базы → не отвечать по сути, честно передать менеджеру.
- **Передача менеджеру:** `handoff_to_manager` (эскалация) или `escalate_to_office`
  (горячий на тест); после авто-хендоффа бот замолкает.
- **Ручной перехват сотрудником:** менеджер жмёт «Перехватить» / отвечает из панели →
  `intercepted=true` (в `DialogState` и карточке) → бот молчит; «Вернуть боту» снимает флаг.

**Реализация «двух слоёв антивранья»:**
- **Слой A (превентивный):** system-промпт `prompts/admission.py` + база `knowledge.py`
  (`ADMISSION_FAQ`, `STOP_WORDS_AND_HOURS`) + детерминированный FAQ-whitelist `core/faq.py`.
- **Слой B (пост-проверка):** `agent/validator.py::validate_reply` — детекторы фактов только
  **логируют** нарушение (код violation в `observ`), автоматически исправляется **только
  markdown** (факты НЕ переписываются). Детекторы прогоняются по варианту текста **до**
  нормализации тире (важный фикс: « — » LLM не должно ослеплять детекторы балла/скидки).

**CRM-адаптер заказчика (`CollegeAdminCrm`, `integrations/crm/college_admin.py`) — не готов:**
методы `create_lead`/`update_stage`/`add_note` шлют POST на `college_admin_webhook_url` с
`Authorization: Bearer <key>`, эндпоинты-**ПРЕДПОЛОЖЕНИЯ**: `/leads`, `/leads/{id}/stage`,
`/leads/{id}/notes`; маппинг стадий → внешние id через `college_stage_map`. Реальные URL,
схема авторизации, формат ответа с id, поведение при дубле номера — **не определено**
(TODO в шапке файла, ждут заказчика).

## 8. Логика автоматического движения карточек

Триггеры перемещения (стадия → колонка через `STAGE_TO_COLUMN`). Все авто-переходы делает
бот в ходе диалога; фоновые джобы двигают только в `follow_up`.

| Событие | Условия | Пред. статус | Новый статус | Доп. действия | Задача/уведомление | Задержка | Отмена менеджером |
|---|---|---|---|---|---|---|---|
| Первое сообщение | номер не пустой, не дубль | — | `greeting` (Новые лиды) | лог входящего; создание карточки | нет | 0 | — |
| Собрано поле квалификации | `ask_qualification` с данными | greeting | `qualification` | `create_lead` при первом поле | нет | 0 | да (drag/перехват) |
| Явный сдвиг | `crm_update_stage(qualification/consulting/test_invite)` | любая | соответствующая | `crm.update_stage` | нет | 0 | да |
| Горячий на тест | `escalate_to_office`, **имя собрано** | qualification | `test_invite` (Приглашён на тест) | `update_stage(test_invite)` | менеджер подтверждает тест | 0 | да |
| Эскалация | `handoff_to_manager` | любая | `manager` (У менеджера) | `escalation_reason`, `intercepted=true`, бот замолкает | карточка у менеджера; awaiting-алерт если клиент ждёт | 0 | «Вернуть боту» |
| FAQ handoff_only | сработало правило (напр. проходной балл, часы работы) | любая | `manager` | `intercepted=true` | как эскалация | 0 | «Вернуть боту» |
| Автодожим | `is_silent` истинно, followup включён, не тихий час | любая (кроме office/manager/follow_up) | `follow_up` (Повторное касание) | 1 пинг клиенту, `followup_sent=true` | нет | ≥ `followup_after_hours` (24ч) | да |
| Ручной перехват | менеджер «Перехватить»/ответ | любая | без смены колонки, `intercepted=true` | `assigned_to`=менеджер | — | 0 | «Вернуть боту» |
| Ручной drag-and-drop | менеджер тащит карточку | любая | выбранная (кроме `silent`) | `COLUMN_TO_STAGE` | — | 0 | да |
| Финальный исход | менеджер жмёт won/lost | любая | outcome фиксируется | защита от авто-перетирания | — | 0 | только менеджер |

- **Вычисляемые колонки** `silent` (и метка «горячий») не хранятся, а считаются на лету
  (`is_silent`, `lead_temperature`) — их нельзя выставить перетаскиванием.
- **Уведомления сотруднику** — не смена карточки, а отдельные WhatsApp-алерты фоновых джоб
  (§10): `awaiting` (клиент ждёт >10 мин), `watchdog` (тишина вебхуков/всплеск ошибок).

## 9. Управление ботом

- **Включение/выключение бота для конкретного диалога:** через `intercepted`
  (`DialogState` + карточка, синхронизируются `_set_intercept`, `router.py:799-807`). Бот
  молчит, если `intercepted=true`.
- **Глобальный и по-ботовый рубильник:** флаги `bots_enabled` (мастер) и
  `bots_enabled:<bot_id>` (переопределяет мастер для одного номера) — переключаются из
  панели (`/admin/flags/{key}`, `/admin/bots/{bot_id}/toggle`), живут в БД (`core/flags.py`),
  меняются **без рестарта**.
- **Ручной перехват сотрудником:** кнопка «Перехватить (бот замолчит)» или отправка ответа
  из панели → `intercepted=true` + `assigned_to`=менеджер (авто).
- **Возврат диалога боту:** кнопка «Вернуть боту» (`/conversation/{user_id}/release`) →
  `intercepted=false` + снятие claim.
- **Временная пауза:** отдельного «pause на N минут» нет; пауза = перехват (`intercepted`).
  При авто-хендоффе бот сам ставит паузу и один раз шлёт `wait_ack`.
- **Защита от одновременного ответа бота и сотрудника:** (а) лок `asyncio.Lock` на ключ
  диалога сериализует ходы; (б) проверка `intercepted` перед ответом; (в) «перехват на лету» —
  перед отправкой перечитывается свежее состояние, если менеджер перехватил во время
  генерации — ответ бота дропается (`orchestrator.py:213-225`).
- **Несколько сообщений подряд:** дебаунс (`debounce_seconds`, по умолчанию **0** = выкл.).
  При включении — быстрые реплики буферизуются, склеиваются и обрабатываются одним ходом LLM
  (без задвоения ответов).
- **Голосовые сообщения:** НЕ распознаются. `kind=non_text` → честный фолбэк
  `NON_TEXT_FALLBACK` («голосовые пока не распознаём, напишите словами или "нужен менеджер"»).
  STT — не реализовано.
- **Изображения/документы:** так же `non_text` → тот же фолбэк. OCR/парсинг — не реализовано.
  (Wappi любой не-`chat` тип сводит к `non_text`; Telegram/Bitrix — по наличию файловых полей.)
- **Повторная отправка вебхуков (Wappi):** дедуп по id события — `_seen_wappi_ids`
  (OrderedDict, максимум **2000**), повтор доставки игнорируется (`main.py:81-94,227`).
- **Дедупликация сообщений (панель):** `add_message` дедупит по `idempotency_key`
  (`store.py`). Дедуп по `provider_msg_id` в самих адаптерах каналов — **нет** (только на
  уровне вебхука Wappi и панели).
- **Секрет вебхука:** `_verify_webhook` (query `s` / заголовок / для Telegram спец-заголовок);
  пустой `webhook_secret` → пропускает (обратная совместимость).

## 10. Напоминания и фоновые действия

Планировщик `scheduler.py`, **тик каждые 300 сек**, 3 джобы (`main.py:48-52`). Всё
конфигурируемо через `config.py`/флаги.

**1. `followup` — автодожим (единственный проактивный пинг клиенту):**
- **Выключен по умолчанию** (`followup_enabled=False`; включается флагом из панели).
- Условие цели: `is_silent` (не перехвачен, дожим ещё не слался, исход не won/lost, колонка
  не office/manager/follow_up, не «шум», есть сообщения, `last_message_at ≤ now −
  followup_after_hours`), канал whatsapp, есть bot_id/chat_id.
- `followup_after_hours=24` (порог тишины). «Шум» — `noise_stale_days=3`.
- **Тихие часы** по Бишкеку (UTC+6, без DST): `followup_quiet_from=22` … `followup_quiet_to=9`
  — в это окно не пишет.
- Действие: **один** двуязычный пинг `FOLLOWUP_PINGS["admission"]` (ru+ky), затем
  `stage=follow_up`, `followup_sent=true` (ровно один пинг за всё время).
- Отмена/override: перехват или `followup_sent` исключают из дожима; флаг можно выключить.

**2. `awaiting` — «клиент ждёт живого менеджера» (алерт команде):**
- Включён по умолчанию (`alerts_enabled=True`), но требует заданных `alert_whatsapp_to` и
  `alert_bot_id` — иначе молчит.
- Условие: диалог передан (stage manager/manager_handoff или `intercepted`) И
  `last_sender==client` (никто не ответил) И исход не won/lost И `last_message_at ≤ now −
  alert_awaiting_minutes` (**10 мин**).
- Действие: WhatsApp-сообщение **команде** (на `alert_whatsapp_to`), НЕ клиенту: «⏳ Клиент …
  ждёт ответа менеджера ~N мин…». Кулдаун `alert_cooldown_minutes=60` (в памяти, сброс при
  рестарте). Карточку не двигает.

**3. `watchdog` — техно-алерты (тишина вебхуков / всплеск ошибок):**
- Включён по умолчанию (`alerts_enabled`), требует `alert_whatsapp_to`+`alert_bot_id`.
- Алерт «тишина»: нет входящих ≥ `alert_silence_minutes` (**30 мин**) → «⚠️ Бот не получал
  сообщений ~N мин. Проверьте Wappi/вебхуки.»
- Алерт «сбои»: прирост (LLM+отправка) ≥ `alert_fail_threshold` (**5**) за тик → «⚠️ N
  сбоев… Проверьте OpenRouter/Wappi.»
- Кулдаун 60 мин на тип. Шлёт на админский WhatsApp. Карточку/клиента не трогает.

**Прочее:**
- **Повторное сообщение клиенту / напоминание о звонке / визите:** отдельных таких напоминаний
  НЕ реализовано (есть только общий `followup`-пинг и `wait_ack` после хендоффа). Напоминание
  «о звонке»/«о визите» с датой — **не определено**.
- **Закрытие неактивного лида:** авто-`lost` НЕТ. Есть архивирование «шума»
  (реклама/медиа-онли/пустой протухший лид, `is_noise`) — вручную кнопкой «Очистить
  рекламу/без диалога» или batch-архив; авто-архива по таймеру нет.
- **Создание задач менеджеру:** отдельной task-сущности нет (см. §3, §8) — «задача» = карточка
  в колонке + алерты.
- **Уведомления сотрудникам:** только через `awaiting`/`watchdog` на **один** настроенный
  номер `alert_whatsapp_to` (не персонально каждому менеджеру).
- **Расписание работы приёмной:** часы работы **не подтверждены** (опросник 0.3 ⚠,
  `COLLEGE_WORKING_HOURS="уточнить у поддержки"`). Тихие часы дожима 22:00–09:00 (Бишкек) —
  это НЕ часы работы приёмной, а окно тишины автодожима.

## 11. Источники рекламы

- **Как определяется источник лида:** в текущем коде — **никак**. Захвата/сохранения источника
  рекламы (Instagram-кампания, referral) **НЕТ**. Поля источника в карточке нет.
- **Передаёт ли Wappi Pro referral-данные:** **не определено** (в `wappi.py` парсятся только
  id/profile_id/from/to/chatId/body/type/chat_type/is_me/senderName/contact_name; поля
  referral/ad не читаются).
- **Как сохраняется реклама/кампания:** не сохраняется (не реализовано).
- **Может ли один клиент прийти из нескольких источников:** **не определено** (модели источника нет).
- **Как строить аналитику по источникам:** не реализовано. `analytics.py` считает
  containment, исходы, по-воронке, по-менеджеру, время ответа/хендоффа, причины эскалации —
  **разреза по источнику нет**.
- **Известные ограничения:** опросник Блок 6 спрашивает «Откуда узнал (Instagram)? ⚠ нужно?»

## 13. Известные edge cases

| Ситуация | Как ведёт себя система сейчас |
|---|---|
| Один номер пишет по нескольким детям | **Не определено.** Ключ диалога — `<bot_id>:<номер>`, одна карточка/квалификация на номер; второй ребёнок затрёт поля. Не решено. |
| Родитель и абитуриент с одного номера | Частично: промпт учит спрашивать «имя будущего студента», если пишет родитель. Отдельных полей «родитель/студент» нет. |
| Меняет выбранную специальность | `direction` перезапишется новым значением (поле изменяемо). Истории смены нет. |
| Сначала отказ, потом возврат | При новом сообщении карточка `archived=false` (всплывает), история сохранена; ручной `lost` не перетирается ботом, но диалог продолжится. |
| Менеджер сменил статус вручную | Ручной `won/lost` защищён от авто-перетирания; drag-and-drop и перехват — источник истины (`_set_intercept` пишет и в `DialogState`, и в карточку). |
| Старое сообщение пришло повторно | Дедуп по id события Wappi (`_seen_wappi_ids`, 2000) + дедуп панели по `idempotency_key`. |
| Несколько сообщений разом | Лок на ключ диалога сериализует; при `debounce_seconds>0` — склейка в один ход. |
| Только голосовое | `non_text` → честный фолбэк (не распознаём), без дебаунса. |
| Прислал документ/фото | `non_text` → тот же фолбэк. Парсинга нет. |
| OpenAI/LLM не ответил | try/except: `record_failure("llm")`, состояние сохраняется, клиент получает мягкий фолбэк «Секундочку, уточню…». Watchdog заметит всплеск. |
| Wappi недоступен при отправке | `record_failure("send")`, сообщение помечается `failed` (в панели «Повторить»), диалог не падает. |
| CRM недоступна | Сбои CRM/панели проглатываются (лог), ответ бота не роняют. `CollegeAdminCrm` без URL — просто warning, пустой id. |
| Неизвестный язык | Спец-обработки нет; LLM отвечает по правилу «язык клиента». Для ru/ky — ок; иное — **не определено**. |
| Клиент пишет ночью | Бот отвечает как обычно (24/7); при эскалации честно предупреждает про рабочее время. Автодожим ночью молчит (тихие часы 22–09). |
| Просит удалить данные | **Не определено.** Механизма удаления ПДн / GDPR-процедуры нет. |

## 14. Нерешённые вопросы

| Вопрос | Почему важен | Возможные варианты | Что сломается, если не решить |
|---|---|---|---|
| Какая модель OpenRouter + `OPENROUTER_API_KEY` | Без модели бот не думает | выбрать модель заказчика; `llm_model_main`/`llm_model_cheap` | Бот не отвечает LLM-логикой (работает только FAQ/fallback) |
| Вебхуки админки заказчика (URL, авторизация, схема лида/стадий, дубли) | `CollegeAdminCrm` — заглушка | получить контракт API у заказчика; заполнить `college_admin_webhook_url`, `college_admin_api_key`, `college_stage_map` | Лиды не уходят во внешнюю CRM (только локальная панель) |
| Доступы WhatsApp-провайдера (Wappi token/profile_id/webhook) | Нет боевого канала | получить креды **своего** аккаунта (не донора) | Прод-запуск в WhatsApp невозможен |
| Заполненный опросник поддержки | База знаний — черновик с ⚠ | поддержка проверяет Блоки 0–8 | Бот эскалирует лишнее / рискует фактами до подтверждения |
| Срок после 11 класса (1г10м vs «3 года») | Спорный факт, бот молчит | подтвердить у поддержки → внести в whitelist | Вопрос всегда уходит менеджеру |
| Цена: за год или за весь курс | Влияет на честность | подтвердить период | «за год/курс?» всегда эскалируется |
| Проходной балл (есть/значение) | Бот не называет цифру | подтвердить порог | Вопрос о балле всегда к менеджеру |
| Формулировка про вуз США | Нельзя обещать | согласовать точный текст | Ограниченный ответ + эскалация |
| Часы работы приёмной | Плейсхолдер «уточнить» | внести факт (напр. Пн–Сб 09:00–18:00) | Вопрос о часах эскалируется; тексты неполны |
| Реальные статусы канбана заказчика (Блок 8) | Текущие колонки — дизайн | снять воронку Bitrix/Trello | Возможное несоответствие процессу заказчика |
| Имя персоны бота (одно/на каждый номер) | Сейчас плейсхолдер «Айдана» | одно имя или три (`manager_name` поддерживает) | Косметика/доверие |
| Захват источника рекламы | Аналитика по кампаниям | завести поле + чтение referral (если Wappi отдаёт) | Нет разреза по источникам |
| Политика ПДн (хранение/удаление) | Личные данные детей | определить срок хранения/удаление по запросу | Юридический риск; запрос «удалите данные» не обработать |
| Задачи менеджеру как сущность | Сейчас «задача» = колонка+алерт | нужен ли трекер задач | Нет явного списка задач |
| Прод-хостинг/домен/деплой | Не развёрнуто | Coolify+nginx (по намёкам в коде) | Нет прод-среды |

## 16. Итоговая схема

```
Входящее сообщение (вебхук Wappi/Telegram/Bitrix)
  → проверка секрета вебхука + дедуп по id события (защита от повторной доставки)
  → определение лида: маршрут к боту (profile_id/BOT_ID) → ключ диалога <bot_id>:<номер>
  → лог входящего в панель (ВСЕГДА, даже при перехвате/выкл. боте)
  → проверка управления ботом: рубильник bots_enabled(:bot_id)? intercepted? → если стоп: молчим
  → [non_text: голос/медиа → честный фолбэк, выход]
  → сохранение сообщения в историю диалога (DialogState.history) + лок на ключ
  → определение намерения: FAQ-whitelist (детерминированно) ИЛИ LLM-воронка admission
  → получение недостающих данных: ask_qualification (база→имя→направление, 1 вопрос/ход)
  → генерация ответа: LLM (промпт+база знаний) → пост-валидатор (детекторы антивранья)
  → обновление карточки: стадия→колонка (STAGE_TO_COLUMN), qualification, outcome, manager_brief
  → установка следующего действия: escalate_to_office / handoff_to_manager (→ бот замолкает)
       либо продолжение диалога; фон: followup/awaiting/watchdog по таймингам
  → ответ в WhatsApp (через канал) → статус доставки ✓/✓✓ обновляется в панели
  → логирование: сбои (observ), нарушения валидатора, аудит панели
```
