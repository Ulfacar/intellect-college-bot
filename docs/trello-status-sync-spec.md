## Решения владельца проекта (утверждено)

> Зафиксировано владельцем проекта. Эти решения **отменяют** соответствующие ПРЕДПОЛОЖЕНИЯ
> ниже и **закрывают** открытые вопросы Q1–Q11. Где семантика помечена ПРЕДПОЛОЖЕНИЕ (смысл
> `pre_contract`/`contract`/финала) — она остаётся до подтверждения заказчиком, но **не
> блокирует** реализацию модели.

- **Q1. Область бота и Trello.** WhatsApp напрямую с Trello не связан. `Trello ↔ Lead`
  (синхронизируется независимо от источника лида); `бот ↔ Conversation`. Бот авто-анализирует и
  двигает **только лиды с активным WhatsApp-Conversation, который ведёт бот**. Карточки из Tilda
  или созданные вручную отображаются в админке и синхронизируются с Trello, но **без диалога бот
  их не классифицирует и не двигает**. Conversation может быть связан с Lead; Lead может
  существовать без Conversation.
- **Q2. Связь Conversation↔Lead.** Отдельные сущности, `conversation.lead_id → lead.id`
  (nullable). При новом WhatsApp-сообщении: (1) искать Lead по **нормализованному номеру**;
  (2) ровно один активный кандидат → связать; (3) не найден → создать новый Lead; (4) несколько
  кандидатов → **авто-связь запрещена**, менеджеру показать выбор; (5) после связи основной
  идентификатор Trello — **`trello_card_id`, не номер**. Один Lead может жить без Conversation.
  **Один номер на нескольких детей — пока не объединять и не перезаписывать, передавать менеджеру.**
- **Q3. Авто-статусы бота.** Бот сам ставит: `new`, `in_progress`, `info_sent`, `callback`,
  `thinking`, `invited`. Условия: `info_sent` — только после **полноценной стандартной**
  консультационной справки (не после приветствия/короткого ответа); `callback` — только явная
  просьба позвонить; `thinking` — только явная фраза «подумаю/обсужу/напишите позже» **до теста**;
  `invited` — только явное согласие прийти/записаться. После `invited`: `lead_status=invited`,
  `bot_phase=handoff`, `dialog_owner=manager`; **дату/время визита подтверждает менеджер**.
- **Q4. `pre_contract` и `contract`.** Два отдельных `lead_status`. «от Альбины»/«список от
  Альбины» **в ключи/логику не зашивать** — ответственный в `assigned_to=albina`. В Trello
  названия списков пока сохраняются как у заказчика. Точный бизнес-смысл — требует подтверждения
  (ПРЕДПОЛОЖЕНИЕ), но не блокирует модель.
- **Q5. Финальный успех.** `contract` предварительно = финальный успешный статус. До подтверждения
  заказчиком: **не** называть «поступил» в аналитике фактом, **не** считать оплатой/зачислением,
  семантику держать как ПРЕДПОЛОЖЕНИЕ. Технически статус реализуем уже сейчас.
- **Q6. `rejected` / `invalid_number`.** На 1-м этапе бот `rejected` **не** ставит авто. ИИ может
  вернуть `intent=possible_rejection` (+`confidence`, `evidence`, `suggested_status=rejected`),
  применяет **менеджер**. `invalid_number` — только менеджер или надёжная техническая проверка
  доставки.
- **Q7. Метки и даты Trello.** Цветные labels пока **не интерпретировать и не синхронизировать**
  по смыслу (до объяснения заказчиком). Даты — через внутренние `next_action_type`
  (`callback`/`visit`/`followup`) + `next_action_at`. Позже `next_action_at` может
  синхронизироваться с Trello due date.
- **Q8. Конфликты.** PostgreSQL — источник текущего состояния; ручное действие из Trello или
  админки = полноценная команда, обновляет PG. Приоритет **ручное > авто**. После ручной смены
  `lead_status`: `manual_status_lock_until = now + 30 минут` — в это окно ИИ анализирует, обновляет
  резюме/температуру, сохраняет `suggested_status`, но `lead_status` авто **не меняет**. Два
  ручных изменения → применяется **последнее по времени получения сервером**. Все конфликты — в аудит.
- **Q9. Доступы Trello.** `api_key`/`token`/`board_id`/`list_id` нужны **только** для фактического
  подключения Trello и **не блокируют**: разделение Conversation/Lead, ввод
  `lead_status`/`bot_phase`/`dialog_owner`, `LeadStatusService`, локальную админку, тесты
  state machine. Секреты — только в `.env`.
- **Q10. Источник лида.** В Lead добавить `lead_source`, `source_campaign`, `source_ad`,
  `source_payload`. `lead_source` — **не колонка и не статус**. Значения: `whatsapp`, `instagram`,
  `tilda`, `manual`, `unknown`. Источник первого обращения ≠ текущий канал
  (пример: `lead_source=tilda`, `conversation.channel=whatsapp`).
- **Q11. Управление ботами и чатами.** Три бота/Wappi-профиля. Работают: общий `bots_enabled`;
  индивидуальные `bots_enabled:<bot_id>`; перехват; возврат боту; ручная отправка →
  `dialog_owner=manager`; перехват **не** меняет `lead_status`; отключение бота не мешает
  приёму/сохранению сообщений; при выключенном боте авто-ответы и авто-смены `lead_status`
  запрещены; ручной Trello-синк продолжает работать. При `dialog_owner=manager`: бот молчит, не
  применяет авто-статусы, допускается фоновая AI-аналитика (summary, temperature, suggested_status).

1. «новые заявки» = заявки с **лендинга Tilda** (`Request from
   [intellect-business-college.tilda.ws]`), НЕ из WhatsApp → **у лидов несколько источников**

3. Огромные накопительные бакеты (отказ 535, инфу 208, неверный 279) → Trello = долгоживущий
   архив; наша админка = живой пер-диалоговый вид. Разная гранулярность.

## 0.5 Зафиксированная архитектура

**Слои (WhatsApp НЕ связан с Trello напрямую):**
```
WhatsApp
  ↕   (Wappi Pro — только транспорт сообщений)
Бот / менеджер
  ↕
Наша админка + PostgreSQL   ← ЕДИНЫЙ ИСТОЧНИК ИСТИНЫ
  ↕   (фоновая двусторонняя синхронизация)
Trello                      ← внешнее отображение карточек и бизнес-статусов
```

**Ответственность компонентов:**
- **Wappi Pro** — только приём/отправка WhatsApp-сообщений.
- **OpenAI/OpenRouter** — понимание диалога, консультация, квалификация, **классификация лида**.
- **Наша админка** — основной рабочий интерфейс менеджеров.
- **PostgreSQL** — источник истины всего состояния.
- **Trello** — синхронизированное внешнее отображение карточек и `lead_status`. НЕ хранит
  переписку и НЕ управляет ботом.
- **Бот НЕ вызывает Trello API напрямую.** Только через внутренние сервисы:
  `AI-классификация → LeadStatusService → PostgreSQL → админка → фоновая синхронизация с Trello`.

**Две отдельные сущности (ключевое решение):** WhatsApp-диалог и карточка Trello — это НЕ
одна сущность.

- **Conversation (диалог):** история сообщений, отправка/приём WhatsApp, управление ботом,
  перехват, кто сейчас отвечает клиенту. Поля (примерно): `conversation_id, channel, phone,
  bot_id, dialog_owner, bot_phase, messages, last_message_at, assigned_to, lead_id`.
- **Lead (карточка клиента):** бизнес-статус, положение в канбане, синхронизация с Trello,
  данные абитуриента, результат. Поля (примерно): `lead_id, name, phone, lead_status,
  lead_source, assigned_to, lead_temperature, qualification, next_action_type, next_action_at,
  trello_card_id, trello_list_id, trello_sync_status`.
- **Связь:** `conversation.lead_id → lead.lead_id`.
- **Допустимые ситуации:** карточка + диалог; карточка из Trello/Tilda **без** диалога; новый
  WhatsApp-диалог создаёт новую карточку или связывается с существующей; Trello синхронизирует
  **карточки лидов, а не сообщения**.

**Три независимых состояния (нельзя одним `stage`):**
- `lead_status` — бизнес-статус карточки (синхронизируется с Trello). 11 ключей (§2).
- `bot_phase` — внутренняя фаза ИИ (greeting/qualification/consultation/waiting/handoff), в
  Trello НЕ уходит.
- `dialog_owner` — кто ведёт диалог (bot/manager/paused). **Перехват меняет только это, НЕ
  двигает колонку.**

Пример: `lead_status=callback`, `bot_phase=handoff`, `dialog_owner=manager` → карточка в
«Перезвонить», диалог у человека, бот молчит.

## 1. Разбор колонок (детально)

Ключи (`lead_status`) **согласованы с заказчиком**; смыслы помечены ПРЕДПОЛОЖЕНИЕ там, где не
следуют прямо из названия колонки.

| Колонка | ключ | Смысл | Ставит | Бот? | Бот отвечает | Финальный |
|---|---|---|---|---|---|---|
| новые заявки | `new` | новая заявка (сейчас — форма Tilda); ПРЕДПОЛОЖЕНИЕ: и новый WA-лид | система/бот | да | да | нет |
| новый пред договор от Альбины | `pre_contract` | ПРЕДПОЛОЖЕНИЕ: преддоговор; «Альбина»→`assigned_to` | менеджер | нет | нет | нет |
| в процессе | `in_progress` | ПРЕДПОЛОЖЕНИЕ: активная консультация | менеджер/бот | да | да | нет |
| Общий договор (список от Альбины) | `contract` | ПРЕДПОЛОЖЕНИЕ: договор оформлен | менеджер | нет | нет | ПРЕДПОЛОЖЕНИЕ да |
| сдали тест, сказали подумают | `tested_thinking` | сдал тест, думает (после теста) | менеджер | нет | ⚠дожим | нет |
| перезвонить | `callback` | нужно перезвонить | менеджер/бот | да | ⚠зависит | нет |
| подумают | `thinking` | клиент думает (до теста) | менеджер/бот | да | да | нет |
| пригласили в college | `invited` | ПРЕДПОЛОЖЕНИЕ: приглашён в колледж/на тест | менеджер/бот | да | нет | нет |
| инфу на вотсап | `info_sent` | ПРЕДПОЛОЖЕНИЕ: отправлена инфо-справка | менеджер/бот | да | да | нет |
| отказ | `rejected` | отказ/слив | менеджер (1-й этап) | нет | нет | да |
| неверный номер | `invalid_number` | ПРЕДПОЛОЖЕНИЕ: ошибочный/недоступный номер | менеджер/система | нет | нет | да |

**Спорные (детальнее):**
- `pre_contract` vs `contract`: ПРЕДПОЛОЖЕНИЕ — преддоговор (бронь/предварительно) vs финальный
  подписанный реестр. Обе «от Альбины» → `assigned_to=albina`, ключ статуса от имени не зависит.
- `thinking` vs `tested_thinking`: **до теста** (бот может, из «я подумаю») vs **после теста**
  (менеджер, нужен результат). Не объединять.
- `info_sent` — главный кандидат на авто-простановку ботом (после **полноценной** инфо-справки,
  не после короткого «адрес/привет»).
- `invited` — пересекается с ботовым «горячий → escalate_to_office»; при простановке ставит
  `bot_phase=handoff`, `dialog_owner=manager` (см. §2 парного документа).

## 3. Предварительная state machine

Переходы: `[подтв.]` следует из данных, `[ПРЕДП.]` — предположение.

Основная ветка (ПРЕДП.):
```
new → in_progress → info_sent → invited → tested_thinking → pre_contract → contract
```
Альтернативы:
```
in_progress/info_sent → callback     (клиент просит звонок)      [бот]
in_progress/info_sent → thinking     (думает до теста)           [бот]
callback/thinking     → in_progress  (клиент вернулся)           [бот]
любая активная        → rejected     (отказ)                     [менеджер]
new/активная          → invalid_number                          [менеджер/система]
```

| Событие | Пред. | Новый | Инициатор | Подтв. менеджера | Диалог | Trello | Админка |
|---|---|---|---|---|---|---|---|
| Новый WA-лид | — | `new` | бот/система | нет | бот ведёт | создать/связать | создать |
| Полная инфо-справка отправлена | new/in_progress | `info_sent` | бот | нет | бот | move | обновить |
| «перезвоните» | активная | `callback` | бот (intent) | нет | без смены владельца | move | обновить |
| «подумаю» (до теста) | активная | `thinking` | бот (intent) | нет | дожим | move | обновить |
| «хочу прийти/запишите» | активная | `invited` | бот (intent) | нет | → handoff, owner=manager | move | обновить |
| Сдал тест, думает | invited | `tested_thinking` | менеджер | — | человек | move | обновить |
| Преддоговор | tested_thinking | `pre_contract` | менеджер | — | человек | move | обновить |
| Договор | pre_contract | `contract` | менеджер | — | человек | move | обновить |
| Отказ | активная | `rejected` | менеджер | да | стоп | move | обновить |
| Неверный номер | new/активная | `invalid_number` | менеджер/система | — | стоп | move | обновить |
| Менеджер тащит карточку | любая | любая | менеджер | — | **owner не меняется** | (источник) | обновить |
| Trello-webhook | любая | новая | Trello | — | **owner/фаза/история не меняются** | (источник) | обновить lead_status |

Финальные: `contract` (ПРЕДП. успех), `rejected`, `invalid_number`.

## 4. Права бота (согласовано, 1-й этап)

**Бот ставит автоматически (только очевидное из текста клиента):** `new`, `in_progress`,
`info_sent` (только после **полной** справки), `callback`, `thinking`, `invited`.

**Только менеджер:** `tested_thinking`, `pre_contract`, `contract`, `rejected` (1-й этап),
`invalid_number`.

**Принцип:** LLM НЕ выбирает колонку. LLM возвращает классификацию → детерминированный
`TransitionService` решает по таблице разрешённых переходов. Подробности AI-классификации,
`suggested_status`, температуры и рекомендаций менеджеру — в §6–§9 парного документа.

Требования: авто-простановка только при `confidence ≥ порог` (ПРЕДП. 0.9); каждое действие
пишет `status_change_source=bot`, `status_change_reason=intent+evidence`; учитывается защитное
окно `manual_status_lock_until` (см. §15 парного документа).

## 5. Двусторонняя синхронизация Trello (тех. дизайн, без кода)

Все изменения `lead_status` проходят через единый **`LeadStatusService`/`TransitionService`**.
Источник истины — PostgreSQL. Trello обновляется фоново через outbox.

### 5.1 Наша админка → Trello
1. Менеджер двигает карточку → бэкенд.
2. `LeadStatusService` валидирует переход, **сразу** пишет `lead_status` в PG,
   `status_change_source=admin`, пишет аудит.
3. Создаётся outbox-событие `trello_sync_status=pending`.
4. Worker перемещает `trello_card_id` в `trello_list_id` цели → `synced`, `last_synced_at`.

### 5.2 Trello → наша админка
1. Trello webhook → эндпоинт.
2. **Дедуп по `action_id`** (уже виден / == `trello_last_action_id` → игнор).
3. Поиск lead по `trello_card_id` → целевой `lead_status` по `list_id`.
4. `LeadStatusService` обновляет `lead_status`, `status_change_source=trello`, сохраняет
   `trello_last_action_id`. **НЕ меняет** `dialog_owner`, `bot_phase`, историю, рубильник бота.
5. Админка (HTMX-опрос) показывает новую колонку.

### 5.3 Бот → оба интерфейса
`intent` (LLM) → маппинг intent→status → `TransitionService` (разрешён ли переход боту +
`confidence`) → PG (`source=bot`) → админка → outbox → Trello.

### 5.4 Защита от цикла (admin → Trello → webhook → снова Trello)
- **Эхо-подавление по `action_id`:** наше исходящее перемещение возвращает `action_id` →
  сохраняем в `trello_last_action_id`; входящий webhook с этим id игнорируется.
- **Сравнение состояния:** webhook с `list_id` == текущему `lead_status` → no-op.
- **Источник:** апдейт `source=trello` НЕ создаёт исходящее Trello-событие (worker шлёт только
  `source ∈ {admin, bot, system}`).
- **Идемпотентный outbox:** одно событие на (card, target_list); повтор схлопывается.

## 7. Работа при сбоях Trello
Админка автономна. При недоступности Trello: `lead_status` обновляется локально сразу;
сообщения менеджеру/клиенту шлются; карточка получает `trello_sync_status=error` +
`trello_sync_error` (индикатор в UI); worker повторяет (ПРЕДП. backoff); **локальный статус не
откатывается**; при восстановлении очередь досинхронизирует идемпотентно (последнее целевое
состояние). Статусы синка: `pending`, `synced`, `error`.
