# Спецификация: управляемая FAQ / база знаний через админку

> **Статус:** черновик на согласование. **Код не меняется этим документом** (без миграций,
> без реализации). Неподтверждённое = **ПРЕДПОЛОЖЕНИЕ**, неизвестное = **не определено**.
>
> **Парные документы:** `docs/admin-bot-control-and-ai-classification-spec.md` (управление
> ботами/классификация), `docs/trello-status-sync-spec.md` (статусы/Trello).
> Ссылки на код — `файл:строка` по текущему состоянию репозитория.

---

## 1. Главный принцип

**FAQ и факты колледжа НЕ должны быть зашиты только в Python-коде.** Менеджер без навыков
программирования должен:
1. изменить ответ в админке,
2. опубликовать его,
3. и бот **сразу** использует новую версию **без перезапуска приложения**.

Следствие: опубликованная база знаний — единый источник фактов и для детерминированного
FAQ-слоя, и для LLM. Хардкод фактов в Python (`knowledge.py`) — переходное состояние, факты
должны приходить из опубликованной базы (см. §7, gap).

---

## 2. Раздел админки «База знаний / FAQ» — что может менеджер

- создавать FAQ;
- редактировать;
- выключать / включать (`is_active`);
- удалять;
- **публиковать** (`is_published`: черновик → опубликовано);
- добавлять варианты формулировок вопроса (`question_variants[]`);
- задавать ответы **на русском и кыргызском** (`answer_ru`, `answer_ky`);
- выбирать категорию (`category`);
- задавать приоритет (`priority`);
- включать `handoff_only`;
- смотреть **историю изменений** (версии);
- **откатывать** предыдущую версию;
- **тестировать вопрос до публикации** (проверка совпадения на черновике).

---

## 3. Поток обработки сообщения

```
Входящее сообщение
 → поиск по АКТИВНЫМ и ОПУБЛИКОВАННЫМ FAQ (is_active AND is_published, срок valid_from/until)
 → есть уверенное совпадение → отправить актуальный ДЕТЕРМИНИРОВАННЫЙ ответ БЕЗ вызова LLM
       → если у правила handoff_only: после ответа передать диалог менеджеру (dialog_owner=manager)
 → совпадения нет → передать вопрос LLM
       → LLM имеет доступ ТОЛЬКО к опубликованной базе знаний
       → при отсутствии подтверждённого факта → эскалация менеджеру
```

**Как сейчас (факт):** `orchestrator._maybe_faq_reply` (`orchestrator.py:296-327`) гоняет FAQ до
LLM; `handoff_only` ставит `stage=manager` + `intercepted=true` (`orchestrator.py:318-320`).
`candidates()` фильтрует по `enabled` и funnel (`faq.py:101-104`). **Отличия от требования:** нет
фильтра `is_published`; LLM берёт факты из хардкода `knowledge.py`, а не из опубликованной базы.

---

## 4. Требование «без перезапуска» (live reload)

- `get_faq_store()` читает из БД при каждом запросе (`faq.py:234-240`) → правки видны **сразу**,
  рестарт не нужен. Это уже так для Postgres-бэкенда.
- `seed_defaults()` (`faq.py:243-481`) при старте **не перетирает** правила, отредактированные
  менеджером (обновляет только `updated_by ∈ {"", "system:seed"}`). Сохранить это правило.
- После введения `is_published`: бот-слой (`candidates`) обязан отдавать **только**
  `is_active AND is_published` и внутри окна `valid_from..valid_until`.

---

## 5. Чувствительные категории (усиленный контроль)

Для категорий: **стоимость, скидки, сроки, проходной балл, оплата, договор** — требуется:
- **подтверждение изменения** перед публикацией (двухшаговый confirm; ПРЕДПОЛОЖЕНИЕ: возможен
  второй утверждающий — политика на согласование);
- **аудит автора** (кто и когда изменил/опубликовал);
- **возможность отката** к предыдущей опубликованной версии.

Причина: это факты, которыми бот рискует «соврать»; они же под детекторами валидатора
(`validator.py`: цена≠6500, скидка %, проходной балл, срок «3 года»). Изменение такого факта в
базе должно быть осознанным и обратимым.

### 6.2 Чего не хватает ❌
| Требование | Пробел |
|---|---|
| `answer_ru` + `answer_ky` | сейчас **один** `answer` (одноязычный) |
| `is_published` (черновик/публикация) | есть только `enabled`=is_active; редактирование = мгновенная публикация, черновика нет |
| Тест **черновика** до публикации | `/faq/test` матчит вход против **живых** правил, не против несохранённого черновика |
| `version` + история + **откат** | версий нет; аудит общий (событие), не снапшоты правила |
| `valid_from` / `valid_until` | нет срока действия |
| **Удаление** (hard delete) | маршрута delete нет — только toggle-выключение |
| Бизнес-`category` (стоимость/скидки/…) | есть только `funnel` (admission/common) |
| Подтверждение для чувствительных категорий | нет двухшагового confirm/утверждения |
| LLM из опубликованной базы | LLM берёт факты из хардкода `knowledge.py`, не из базы |

## 7. Предлагаемая модель данных

**Минимальная модель FAQ (по требованию + совместимость):**

| Поле | Тип | Комментарий / текущее соответствие |
|---|---|---|
| `id` | id | есть |
| `category` | enum/строка | **новое**: general, address, hours, directions, test, documents, deadline, **cost**, **discount**, **duration**, **passing_score**, **payment**, **contract**, usa (чувствительные — жирным, §5) |
| `funnel`/`scope` | enum | оставить (admission / common) — область воронки, отдельно от `category` |
| `title` | строка | есть |
| `question_variants[]` | list | **переименовать** из `patterns[]` |
| `negative_terms[]` | list | оставить (точность матчинга; не в мин.модели, но полезно) |
| `answer_ru` | текст | **разделить** из `answer` |
| `answer_ky` | текст | **новое** |
| `is_active` | bool | = текущий `enabled` |
| `is_published` | bool | **новое** (черновик/опубликовано) |
| `priority` | int | есть |
| `handoff_only` | bool | есть |
| `allow_during_qualification` | bool | оставить (есть) |
| `valid_from` | timestamp/null | **новое** |
| `valid_until` | timestamp/null | **новое** |
| `updated_by` | строка | есть |
| `updated_at` | timestamp | есть |
| `version` | int | **новое** (инкремент при публикации) |

**Таблица версий (новое)** `faq_version`: `id, faq_id, version, category, title, question_variants,
answer_ru, answer_ky, priority, handoff_only, valid_from, valid_until, published_by, published_at`
— снапшот каждой опубликованной версии для истории и отката.

**Правило матчинга бота:** `candidates()` отдаёт только `is_active AND is_published` и
`valid_from ≤ now ≤ valid_until` (null-границы = без ограничения). Черновики (`is_published=false`)
бот **никогда** не отдаёт.

**Язык ответа:** выбор `answer_ru`/`answer_ky` по определённому языку клиента (механизм детекта
из `common.py`); при отсутствии перевода — ПРЕДПОЛОЖЕНИЕ: фолбэк на доступный язык + пометка
менеджеру «нет перевода».

## 8. UI раздела «База знаний / FAQ»

**Список правил (таблица):** категория, заголовок, приоритет, статус (черновик/опубликовано,
активно/выключено), срок действия, версия, автор/дата, `handoff_only`. Фильтры: по категории,
по статусу (черновик/опубликовано), по funnel, поиск по тексту. Бейдж «⚠ чувствительная
категория».

**Форма правила:** заголовок; категория (селект); варианты вопроса (мультиввод); `answer_ru`;
`answer_ky`; приоритет; `handoff_only`; `allow_during_qualification`; `valid_from`/`valid_until`;
переключатель активности.

**Действия:** «Сохранить черновик», «Тест» (проверить совпадение и предпросмотр ответа на
черновике, до публикации), «Опубликовать» (для чувствительных — шаг подтверждения), «История
версий» (список версий + «Откатить к версии N»), «Выключить/Включить», «Удалить».

**Тест-панель:** ввод примерного вопроса клиента → показывает, какое правило сматчится (или
«уйдёт в LLM/эскалацию»), с учётом приоритетов/negative_terms; для черновика — против
несохранённой версии.

> Текущий `faq.html` уже даёт список+форму+тест по funnel; добавляются: колонки/переключатель
> публикации, поля `answer_ky`/`category`/срок, история/откат, шаг подтверждения для §5.

## 9. Правила публикации и отката

**Публикация:**
- редактирование меняет **черновик** (`is_published=false`), бот его не видит;
- «Опубликовать» → `is_published=true`, `version += 1`, снапшот в `faq_version`, запись в аудит
  (`published_by/at`);
- бот сразу использует опубликованную версию (live, без рестарта);
- **чувствительные категории** (§5): публикация только после явного подтверждения
  (ПРЕДПОЛОЖЕНИЕ: возможно второй утверждающий) — иначе остаётся черновиком.

**Откат:**
- «Откатить к версии N» → создаёт новую опубликованную версию из снапшота N (`version += 1`,
  не переписывает историю), аудит «rollback to N by <login>»;
- бот сразу отдаёт откатанную версию.

**Конфликты/приоритет:** ручное действие менеджера в аудите; при равном приоритете двух
совпадений — как сейчас, `match_faq` возвращает None (не угадывает, `faq.py:69-72`).

**Согласованность с антивраньём:** факты чувствительных категорий, публикуемые через базу,
должны соответствовать детекторам `validator.py` (напр. цена 6500). ПРЕДПОЛОЖЕНИЕ: при
публикации — мягкая проверка ответа валидатором и предупреждение автору (не блок).

## 10. Необходимые тесты (без реализации)

**Модель/хранилище**
- CRUD: create/edit/delete; toggle активности; память и Postgres дают одинаковый результат;
- `version` инкрементится при публикации; `faq_version` хранит снапшоты; история читается.

**Публикация/бот-видимость**
- бот отдаёт **только** `is_active AND is_published` в окне `valid_from..valid_until`;
- **черновик не обслуживается** ботом;
- редактирование → публикация → следующее сообщение использует новую версию **без рестарта**;
- истёкший `valid_until` → правило не отдаётся (уходит в LLM/эскалацию).

**Откат**
- «Откатить к версии N» → бот отдаёт версию N; история и аудит содержат событие rollback.

**Язык**
- `answer_ru` для ru, `answer_ky` для ky; отсутствие перевода → фолбэк + пометка.

**Чувствительные категории**
- публикация в cost/discount/duration/passing_score/payment/contract **требует подтверждения**;
  без подтверждения остаётся черновиком; аудит автора записан;
- (опц.) предупреждение валидатора при несоответствии факта (напр. цена ≠ 6500).

**Поток обработки**
- `handoff_only` правило: после ответа `dialog_owner=manager` (перехват);
- нет совпадения → путь LLM; LLM оперирует опубликованной базой; нет факта → эскалация.

**Регресс**
- `seed_defaults` не перетирает опубликованные ручные правила; существующие 136 тестов зелёные.
