# План: MVP бота колледжа (Opus → Fable design → Codex code → Fable review)

Роль этого документа — **план** (venom mode, автор Opus). Из него Fable делает детальный
дизайн (контракты, UX диалога, UI админки), Codex кодит по дизайну, Fable ревьюит.
План задаёт scope, решения и критерии приёмки — НЕ финальные тексты/интерфейсы (это дизайн Fable).

## 1. Цель MVP

Бот-консультант приёмной комиссии Intellect IT & Business College в WhatsApp: отвечает 24/7
на языке клиента (ru/ky), **только фактами из базы знаний**, при неуверенности — эскалация на
менеджера; заводит/двигает заявку абитуриента в канбане; автодожим долго думающих.
Гоняется в Telegram-песочнице до получения WhatsApp-доступов.

## 2. Ключевые решения (зафиксированы)

1. **Одна воронка `admission`** вместо tours/visa/tickets. Направление колледжа — атрибут лида, не отдельная воронка.
2. **3 номера = один мозг.** Реестр `BOTS`, scenario=`admission` у всех, маршрут по `wappi_profile_id`.
3. **Языки:** авто-детект ru/ky, ответ на языке клиента (механизм из `common.py` переиспользуется).
4. **Никогда не врёт — 2 слоя** (переносим из каркаса):
   - Слой A: промпт + FAQ-whitelist (`prompts/admission.py` + `knowledge.py` + `core/faq.py`).
   - Слой B: пост-валидатор `validator.py` — детекторы под запреты колледжа.
5. **Факты фиксированные** — БЕЗ live-поиска (TourVisor вырезаем). Цена $6500 называется как
   официальный факт (аналог бывш. VISA_SERVICE_PRICES); размер скидки — НЕ называется (после теста → эскалация).
6. **CRM = админка Emir через вебхуки.** Новый адаптер `CollegeAdminCrm(CRMPort)` по образцу `bitrix24.py`.
   До получения вебхуков — `CRM_BACKEND=stub` (MVP работает без реальной CRM).

## 4. Whitelist фактов (источник правды для T2)

Берётся из `docs/опросник-для-поддержки.md` (после заполнения поддержкой) и текущего черновика
`memory/college-domain-facts`: приём после 9/11, дедлайн 12.08, срок 2г10м/1г10м(⚠), $6500,
скидка после теста, тест матем+англ 1.5ч (порог 90+ ⚠), 8 направлений (IT+бизнес), документы
(свид-во о рождении/паспорт), адрес Ибраимова 103/1А, US-поток. **Спорные факты бот не утверждает** — при вопросе эскалация.

## 5. Границы «сам / → менеджер» (для дизайна UX диалога)

- **Сам:** приветствие, ответы из whitelist, квалификация (1 вопрос за ход), приглашение на тест/в колледж, языки ru/ky.
- **→ Менеджер (эскалация):** вопроса нет в whitelist; торг о цене/индивидуальная скидка; оплата/договор; жалоба/агрессия; клиент просит человека; клиент «горячий» (готов оформляться/на тест); бот не уверен; 2 хода без осмысленного ответа.
- **Никогда:** гарантия поступления/зачисления/гранта; выдуманные баллы/сроки/суммы; ответ по теме вне базы знаний.

## 7. Распределение по конвейеру

- **Fable (design):** (a) техдизайн — сигнатуры `AdmissionFunnel`, exec_tool, `CollegeAdminCrm`, маппинг стадий→колонки канбана; (b) UX диалога — сценарии/тон/скрипты приветствия и эскалации на ru/ky; (c) UI админки — канбан статусов абитуриентов, чат, перехват.
- **Codex (code):** реализация T1–T8 по дизайну Fable.
- **Fable (review):** ревью дифа Codex против этого плана и правила «не врать» (агент `codex-reviewer` + fable).

## 8. Блокеры (не мешают T1–T6, T8)
- Вебхуки админки → T (CRM-адаптер, отдельно, Фаза 2).
- Модель OpenRouter → T7 (плейсхолдер сейчас).
- WhatsApp-доступы → прод-запуск (MVP гоняем в Telegram).

## 9. Acceptance MVP
Бот в Telegram-песочнице: отвечает на 10 частых вопросов из whitelist (ru и ky), корректно
эскалирует на 6 триггеров из §5, заводит карточку в stub-канбане, не выдаёт ни одного факта вне
whitelist (проверка валидатором + ручной прогон сценариев).
